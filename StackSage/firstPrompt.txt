## PROJECT OVERVIEW: StackSage

StackSage is a RAG-powered web application that helps developers plan and build any project intelligently. A user inputs:
- A **project idea** (text description)
- Their **hardware specifications** (RAM, GPU VRAM, CPU, OS)

The platform then performs three core functions:

### Function 1 — Project Blueprint
Generate a **complete, detailed guide** on how to build the project:
- Step-by-step architecture explanation
- An **interactive visual workflow/graph** of the recommended tech stack and AI tools
- Node icons and labels for each tool in the graph
- Hardware-aware filtering: clearly mark which tools/models are **compatible vs. incompatible** with the user's hardware
- Always include **free/open-source alternatives** alongside paid options
- Powered by a live RAG pipeline that stays updated with the latest AI tooling

### Function 2 — Agent Implementation Prompt
Generate a **comprehensive, copy-paste-ready prompt** the user can give to a coding agent (like Claude Code or Cursor). This prompt should:
- Describe the full project scope
- Include architecture decisions
- Specify which tools/libraries to use
- Include file structure expectations
- Be detailed enough to maximize how much of the project the agent can implement autonomously

### Function 3 — Repo-to-Text Converter
A utility that:
- Accepts a **GitHub repo URL** or a **local file upload**
- Converts the entire repository into a single structured text blob (similar to repomix/gitingest)
- Formats it so it can be pasted into an LLM prompt for maximum context injection
- Shows an estimated token count so the user knows if it fits their target LLM's context window
- Helps users get dramatically better answers from AI tools by giving them full project context

## 1. HARDWARE: 8GB RAM, NO GPU, LIMITED DISK SPACE

The developer is working on a standard HP laptop with:
- **8GB RAM total**
- **No dedicated GPU / No CUDA**
- **~6GB free disk space on the primary drive (C:)**
- **OS: Windows**

### What this means for every decision you make:

**No local LLM inference.**
Do not suggest, use, or install Ollama, llama.cpp, LM Studio, or any tool that runs a model locally. These require 6–16GB RAM minimum and will either crash the system or make it unusable. All LLM calls must go to a free cloud API.

**No GPU-accelerated anything.**
Do not use CUDA, torch with GPU, or any library that requires a GPU driver. If PyTorch is needed for any reason, install the CPU-only version explicitly: `pip install torch --index-url https://download.pytorch.org/whl/cpu`

**Disk space is critical.**
- Docker images and volumes will consume disk space fast. Be explicit about which images are being pulled and their approximate sizes.
- Do NOT pull large base images. Use slim/alpine variants wherever possible (e.g. `python:3.11-slim`, `node:20-alpine`).
- The Qdrant Docker image is ~200MB — acceptable.
- Do NOT pull any model weights locally under any circumstance.
- Warn the developer before any `docker pull` or `pip install` that might consume significant space.
- Total Docker image footprint must stay under 3GB combined.

**Development workflow — services run individually, Docker comes last.**
- During development, run each service directly (uvicorn, redis-server, celery) rather than via Docker Compose. This avoids Docker overhead eating into the 8GB RAM.
- Docker Compose is reserved for the final containerization phase only.
- Write the `docker-compose.yml` early so it is ready, but instruct the developer NOT to run `docker compose up` until Phase 7 (final integration).
- For local dev, provide a `dev.sh` (or `dev.bat` for Windows) script that starts all services individually in separate terminals.

**Qdrant — use Qdrant Cloud free tier during development.**
- Do NOT run Qdrant locally in Docker during development on this machine.
- Instead, use **Qdrant Cloud** (`cloud.qdrant.io`) — free tier gives 1 cluster with 1GB storage, zero RAM cost on the developer's machine.
- Required env vars: `QDRANT_URL` and `QDRANT_API_KEY` (both provided by Qdrant Cloud dashboard).
- When writing the final Docker Compose for deployment, include a local Qdrant container as an option but keep Qdrant Cloud as the default via env var.

**Redis — run natively during development.**
- Install Redis natively: on Windows use `winget install Redis.Redis` or run via WSL2.
- Do not run Redis in Docker during development.
- Redis memory usage is minimal (~50MB) so this is fine.

**RAM budget during development (must stay within 8GB):**

| Service | Approx RAM |
|---|---|
| Windows OS baseline | ~2.5GB |
| VS Code + terminal | ~400MB |
